# ADR-009

---

## Título

Selección de un medio para almacenar las fotos de eventos de estacionamiento.

## Fecha

24/03/2026

## Estado

Tomada

---

## Contexto

El sistema captura fotos del estacionamiento asociadas con cada evento de cambio de estado. Estas fotos deben almacenarse de forma persistente y estar accesibles para revisión histórica. Almacenar datos binarios en PostgreSQL es posible pero poco escalable. Se necesita una solución de almacenamiento separada para las imágenes.

## Decisión

Almacenar todas las fotos de eventos de estacionamiento en **Amazon S3**, organizado como un data lake. Cada foto se almacena con una clave que referencia el ID del evento correspondiente en PostgreSQL (`events/{event_id}/photo.jpg`). El componente `Persistencia` gestiona todas las interacciones con S3.

---

## Factores decisivos

1. S3 está diseñado específicamente para almacenamiento de objetos y es mucho más rentable y eficiente para estos que almacenar arrays binarios en PostgreSQL.  
2. S3 provee 11 nueves de durabilidad, lo cual es más que apropiado para registros fotográficos históricos.  
3. Las políticas de ciclo de vida de S3 pueden usarse para archivar o eliminar automáticamente fotos antiguas, gestionando los costos de almacenamiento a largo plazo.

## Alternativas

- **Almacenar fotos en PostgreSQL como binarios**: Simple desde la perspectiva de desarrollo, pero incrementa el tamaño de la DB exponencialmente, degrada el rendimiento de las consultas, y no es la forma en que se deben usar las bases de datos relacionales.  
- **Almacenar fotos en el volumen EBS de EC2**: Atado a una única instancia, no persistente ante reemplazos de instancia, y no escala independientemente del backend de lógica. Descartado.  
- **Almacenamiento de terceros (por ejemplo, Cloudinary)**: Agrega dependencia externa y costo. S3 ya está en el ecosistema de AWS que se usa para el resto del backend.

## Pros y contras de S3 como almacenamiento de fotos

**Pros:**

- Almacenamiento de objetos rentable y altamente durable.  
- Desacoplado de la base de datos y del backend de lógica, escalando independientemente.

**Contras:**

- Agrega dependencia de la disponibilidad de S3 para la recuperación de fotos.

---

## Consecuencias

El componente Persistencia debe implementar la siguiente lógica de escritura: primero escribir la foto en S3, luego insertar el registro del evento en PostgreSQL con la clave de S3. Las fallas parciales (foto subida pero falla la inserción en DB) deben manejarse con un mecanismo de limpieza o reintento.
