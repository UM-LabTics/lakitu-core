# ADR-007

---

## Título

Gestión óptima del estado del estacionamiento en tiempo real.

## Fecha

24/03/2026

## Estado

Tomada

---

## Contexto

El frontend web necesita mostrar el estado actual del estacionamiento en tiempo casi real. Cada evento de cambio de estado desde el Pi actualiza el estado completo del estacionamiento. Este estado es consultado frecuentemente por los clientes conectados y debe servirse con latencia muy baja. Consultar la base de datos en cada lectura sería lento y generaría carga innecesaria en la misma.

## Decisión

Usar **Redis** como caché en memoria dentro de la capa de Lógica de Negocio para almacenar el estado actual del estacionamiento. Cuando llega un nuevo evento desde el Pi, el Cloud Receptor actualiza Redis con el nuevo estado. El Cloud Backend lee de Redis para responder las consultas de los clientes sobre el estado actual.

---

## Factores decisivos

1. Redis está diseñado específicamente para acceso de baja latencia a pares clave-valor, haciéndolo ideal para servir el estado actual (una única estructura de datos) a muchos clientes concurrentes.  
2. Separar la preocupación del "estado actual" de la base de datos persistente reduce la carga sobre esta y mantiene el camino de lectura simple y rápido.  
3. El estado actual del estacionamiento es un payload pequeño; Redis está perfectamente dimensionado para este caso de uso.

## Alternativas

- **Consultar la base de datos directamente para el estado actual**: Simple, pero introduce latencia y carga de base de datos en cada lectura. A medida que crece el número de clientes conectados, esto se convierte en un cuello de botella.  
- **Estado en memoria de Cloud Backend**: Almacenar el estado actual en la memoria de la aplicación. Rápido, pero no durable ante reinicios, y no es compartible con tener múltiples instancias del proceso (por si se deseara escalar horizontalmente en un futuro).

## Pros y contras de Redis como caché de estado

**Pros:**

- Latencia de lectura en milisegundos para consultas del estado actual.  
- Reduce la carga sobre la base de datos significativamente.  
- Modelo de datos simple (el estado del estacionamiento puede almacenarse como un hash de Redis o string JSON).  
- Librerías cliente bien soportadas en todos los lenguajes de backend principales.

**Contras:**

- Agrega un componente de infraestructura adicional para desplegar y gestionar.  
- Los datos de Redis se pierden al reiniciar a menos que se configure persistencia; para el caché de estado actual esto es tolerable ya que el Pi re-emitirá el estado actual al reconectarse.  
- Introduce una posibilidad de inconsistencia entre el caché de Redis y la base de datos si las actualizaciones no se manejan bien.

---

## Consecuencias

El camino de escritura para un nuevo evento de estacionamiento debe actualizar Redis primero (para servicio inmediato a los clientes) y luego almacenarse en la base de datos de forma asíncrona.

## Comentarios

Redis se ubica en la capa de Lógica de Negocio en lugar de la capa de Datos porque sirve como caché operacional gestionado y utilizado únicamente por un componente para cumplir con su lógica de negocio, no como almacenamiento durable. Esta distinción se refleja en el diagrama de arquitectura.
