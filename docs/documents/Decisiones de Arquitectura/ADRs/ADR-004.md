# ADR-004

---

## Título

Cómo comunicar al backend el estado actual del estacionamiento.

## Fecha

16/03/2026

## Estado

Tomada

---

## Contexto

Cuando un lugar de estacionamiento cambia su ocupación, el sistema debe comunicar el nuevo estado al cloud. Se requiere decidir si el Pi enviará todo el estado del estacionamiento, o solo las partes que hayan cambiado.

## Decisión

En cada evento de cambio de ocupación, el **Raspberry Pi envía el estado completo actual** del estacionamiento entero (todos los lugares y sus estados) como un único payload JSON, en lugar de solo los lugares que cambiaron.

---

## Factores decisivos

1. El estacionamiento monitoreado es pequeño, lo que significa que el payload del estado completo es pequeño y el costo de ancho de banda de enviar el estado completo es despreciable.  
2. Enviar el estado completo elimina el riesgo de que el modelo del cloud diverja del modelo del Pi debido a mensajes perdidos o fuera de orden. El estado del cloud simplemente se sobreescribe con el estado del Pi en cada evento.  
3. La gestión de actualizaciones constantes a partes específicas del estado, mientras se mantiene el resto intacto involucra una complejidad muy superior, sobre todo en períodos pico (como las 8am) cuando pueden detectarse varios cambios por minuto. La lógica del cloud receptor se ve significativamente simplificada.

## Alternativas

- **Solo actualizaciones de variaciones**: Enviar solo el/los lugar(es) que cambiaron por evento. Más eficiente en ancho de banda, pero introduce riesgo de consistencia de estado y requiere que el cloud mantenga un modelo de estado completo que pueda actualizarse incrementalmente. Para un estacionamiento pequeño, el ahorro de ancho de banda es despreciable.  
- **Actualizaciones de variaciones con sincronización periódica del estado completo**: Enfoque híbrido. Más complejo sin beneficio significativo a esta escala.

## Pros y contras del payload de estado completo por evento

**Pros:**

- Garantiza consistencia eventual: incluso si mensajes anteriores se perdieron, el siguiente evento resincroniza completamente el estado del cloud.  
- Simplifica significativamente el Cloud Receptor y la lógica de actualización de estado actual.  
- Sencillo de depurar: cada mensaje es autocontenido y completamente interpretable sin historial.

**Contras:**

- Tamaño de mensaje ligeramente mayor por evento (aunque despreciable para un estacionamiento pequeño).  
- La base de datos almacena datos redundantes (estados de lugares sin cambios) con cada registro de evento, incrementando ligeramente el uso de almacenamiento.  
- El evento almacenado en la base de datos representa un snapshot completo del estacionamiento en un punto en el tiempo, no solo un cambio. Esto hace que las consultas históricas sean más simples (cada fila es un estado completo).
