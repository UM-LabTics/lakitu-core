# ADR-010

---

## Título

Definición de el/los medio(s) de comunicación entre el sistema y los usuarios finales.

## Fecha

16/03/2026

## Estado

Tomada

---

## Contexto

El frontend tiene dos patrones de acceso a datos claramente distintos: (1) consumir el estado actual y en vivo del estacionamiento a medida que cambia en tiempo real, y (2) consultar datos históricos de ocupación. Estos dos tipos de peticiones tienen características fundamentalmente diferentes en términos de latencia, frecuencia de transmisión, duración de la conexión y volumen de datos.

## Decisión

Exponer **dos APIs separadas desde el backend**: una **API WebSocket** para actualizaciones del estado del estacionamiento en tiempo real y solicitudes de fotos bajo demanda, y una **API REST** para consultas de eventos históricos y operaciones administrativas. Ambas APIs comparten el mismo componente Auth para la autenticación.

---

## Factores decisivos

1. La consulta del estado actual del estacionamiento está centrado en la idea de notificaciones (no push) en tiempo real: el backend debe avisar a todos los clientes conectados inmediatamente cuando ocurre un cambio de estado (comunicación servidor a cliente). El otro tipo de consulta en tiempo real es el de sacara fotos a demanda, comunicación que debe ocurrir en el sentido opuesto (cliente a servidor). WebSocket es el protocolo natural para estos requisitos.  
2. Las consultas históricas son operaciones de petición y respuesta sin requisito de tiempo real. REST está bien adaptado, es ampliamente entendido y usado, fácil de documentar y testear.  
3. Mezclar semánticas de tiempo real y request-response en un único protocolo, ya sea implementar consultas históricas con WebSockets o comunicación en tiempo real con REST, complicaría innecesariamente tanto la implementación del cliente como la del servidor.  
4. Separar las APIs permite que cada una sea diseñada y optimizada para su propio tipo de acceso.

## Alternativas

- **Solo WebSocket (todas las consultas sobre WebSocket)**: Técnicamente viable, pero fuerza un patrón request-response sobre una conexión bidireccional, requiriendo un incrementando significativo de la complejidad del cliente.  
- **Solo REST con consultas periódicas**: El cliente consulta periódicamente un endpoint REST para el estado actual. Introduce latencia, desperdicia mucho el ancho de banda, y es una experiencia de usuario inferior comparada con server-push vía WebSocket.  
- **Eventos para tiempo real, REST para histórico**: Es más simple que WebSocket para push unidireccional, pero no soporta la comunicación bidireccional necesaria para las solicitudes de fotos a demanda. Dos canales serían necesarios, haciendo el desarrollo más complejo.  
- **WebSocket para tiempo real, GraphQL para histórico**: Opción muy atractiva, GraphQL simplifica enormemente las consultas a datos relacionales estructurados, sin embargo sus beneficios son más aparentes en sistemas con bases de datos más complejas que la que trataremos nosotros. Descartado por aportar beneficio limitado para un sistema con pocos tipos de datos, como el nuestro, y por la ligeramente superior complejidad de definición y documentación que puede agregar.

## Pros y contras de la estrategia dual WebSocket \+ REST

**Pros:**

- Cada API está diseñada exactamente para su patrón de acceso, manteniendo ambas implementaciones limpias y centradas en su responsabilidad.  
- La API REST es trivial de testear con herramientas estándar (como Postman) y documentar.  
- El ciclo de vida de la conexión WebSocket (conectar, recibir pushes, enviar comandos) está bien definido y es simple de implementar para el frontend.  
- El componente Auth compartido evita duplicar la lógica de autenticación.

**Contras:**

- El cliente del frontend debe gestionar dos tipos de conexión diferentes.  
- Ambas APIs deben mantenerse y versionarse independientemente.

---

## Consecuencias

El Cloud Backend debe implementar un registro de conexiones WebSocket para rastrear todos los clientes actualmente conectados y transmitir eventos de cambio de estado a todos ellos cuando llega un nuevo evento desde el Pi. La API REST debe definir semánticas claras de paginación y filtrado para las consultas de eventos históricos, y en general seguir buenas prácticas típicas.
