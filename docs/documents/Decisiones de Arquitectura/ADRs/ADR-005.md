# ADR-005

---

## Título

Selección de un medio para la comunicación entre el Raspberry Pi y el servidor cloud.

## Fecha

24/03/2026

## Estado

Tomada

---

## Contexto

El Raspberry Pi debe enviar de forma confiable los eventos de cambio de estado del estacionamiento al backend cloud. Estos son payloads JSON pequeños y generalmente infrecuentes, disparados por cambios de estado. El canal de comunicación debe manejar: conectividad intermitente, autenticación segura del dispositivo, ingesta de mensajes escalable, y la capacidad de que el servidor cloud envíe comandos de vuelta al Pi (sacar una foto a demanda).

## Decisión

Usar **AWS IoT Core** como proveedor de comunicación mediante MQTT para conectar el Pi a cloud y cloud al Pi. Una **IoT Core Rule** reenvía los mensajes entrantes del Pi a una **cola SQS**, que el componente Cloud Receptor en EC2 consume. Para las solicitudes de fotos bajo demanda, el Cloud Receptor publica directamente en un topic de IoT Core al que el Pi está suscripto.

---

## Factores decisivos

1. IoT Core soporta nativamente MQTT, que es liviano y está bien adaptado para hardware embebido como el Raspberry Pi operando sobre redes potencialmente limitadas.  
2. IoT Core gestiona la autenticación y autorización del dispositivo mediante certificados X.509, eliminando la necesidad de gestionar credenciales.  
3. SQS provee una cola de mensajes confiable y ordenada con entrega garantizada, desacoplando la tasa de publicación del Pi de la tasa de consumo del Cloud Receptor.  
4. El Rule Engine de IoT Core permite enrutar mensajes a SQS (y potencialmente otros servicios) sin código personalizado, simplificando la programación de la ingesta.  
5. La naturaleza bidireccional de MQTT (los topics pueden ser tanto publicados como suscritos) soporta nativamente tanto el envío de eventos como la recepción de comandos en una única conexión.

## Alternativas

- **HTTPS directo al backend**: El Pi podría enviar solicitudes POST directamente a un endpoint REST en el Cloud Receptor. Descartado porque requiere que parte del backend esté expuesta públicamente, carece de gestión de identidad de dispositivos integrada, y es carente ante falta de disponibilidad temporal del lado cloud.  
- **AWS Kinesis Data Streams**: Diseñado para streams de datos de alto volumen y continuos. Excesivo para un sistema que genera eventos discretos e infrecuentes, y añade costo y complejidad innecesarios.  
- **WebSockets directo al backend**: Soportaría comunicación bidireccional pero requiere mantener conexiones persistentes, manejar lógica de reconexión y de autenticación del dispositivo, y es operacionalmente más pesado que MQTT para un dispositivo embebido.

## Pros y contras de IoT Core \+ SQS

**Pros:**

- Comunicación MQTT ya gestionada y escalable sin servidor ni servicio que mantener.  
- La autenticación de dispositivos mediante certificados X.509 es segura y está desacoplada del resto de la aplicación.  
- SQS funciona como buffer de eventos si el Cloud Receptor está temporalmente caído o lento, asegurando que no se pierdan eventos.  
- Los topics MQTT bidireccionales sirven tanto para el pipeline de eventos como para el flujo de comandos de foto bajo demanda.  
- El Rule Engine de IoT Core desacopla el enrutamiento de mensajes de la lógica de la aplicación.

**Contras:**

- Introduce dependencias específicas de AWS; migrar fuera del mismo sería muy complejo.  
- Agrega latencia respecto a una conexión directa.
