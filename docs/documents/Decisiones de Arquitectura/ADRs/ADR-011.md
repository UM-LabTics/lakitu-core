# ADR-011

---

## Título

Despliegue del backend.

## Fecha

24/03/2026

## Estado

Tomada

---

## Contexto

Los componentes del backend cloud (Cloud Receptor, Cloud Backend, Auth, Persistencia) deben ejecutarse en algún lugar de AWS. La arquitectura soporta conexiones WebSocket persistentes para actualizaciones en tiempo real, un consumidor de SQS, y procesos que mantienen estado en memoria vinculado a Redis. Se debe elegir una plataforma de cómputo.

## Decisión

Ejecutar todos los componentes del backend cloud en una **única instancia EC2** (t3.small o t3.medium), con componentes desplegados como procesos separados o contenedores Docker en el mismo host.

---

## Factores decisivos

1. Las conexiones WebSocket persistentes requieren un proceso servidor de larga ejecución, lo cual es incompatible con el modelo de ejecución serverless y de corta duración de AWS Lambda.  
2. El consumidor de SQS (Cloud Receptor) es un loop que también requiere un proceso persistente.  
3. Una única instancia EC2 es operacionalmente simple para un equipo pequeño, evitando la sobrecarga de orquestar múltiples servicios.

## Alternativas

- **AWS Lambda (serverless)**: Rentable y escala automáticamente, pero fundamentalmente incompatible con conexiones WebSocket persistentes y loops consumidores de SQS con estado. Requeriría la integración WebSocket de AWS API Gateway, añadiendo complejidad innecesaria.  
- **AWS Elastic Beanstalk**: Despliegue basado en EC2 gestionado. Más fácil que EC2 crudo, pero introduce abstracción adicional que puede ser más difícil de depurar. EC2 crudo da al equipo más control directo y valor de aprendizaje.  
- **Múltiples EC2, separando por servicio**: Permite desacoplar físicamente los componentes, pero agrega complejidad operacional y de desarrollo excesiva. La gestión de múltiples máquinas, con sus diferentes interfaces de comunicación y la latencia que esta comunicación agregaría son demasiadas desventajas a cambio de poca ganancia.

## Pros y contras de una única instancia EC2

**Pros:**

- Simple de usar, acceder por SSH y gestionar.  
- Soporta todos los patrones requeridos (procesos persistentes, loops de petición a SQS, comunicación WebSocket).  
- Nos permite aprender sobre administración de servidores Linux.

**Contras:**

- Punto único de falla; si la instancia cae, todos los componentes del backend quedan no disponibles.  
- Sin escalado automático; requeriría intervención manual ante carga inesperada.
