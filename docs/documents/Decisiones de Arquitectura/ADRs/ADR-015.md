# ADR-015

---

## Título

Selección de una librería de autenticación y una librería de hashing de contraseñas para el componente `Autenticación` del backend.

## Fecha

11/04/2026

## Estado

Tomada

---

## Contexto

Dado que la arquitectura define un componente de autenticación, utilizado tanto por la API REST como por la API WebSockets, es necesario decidir cómo se generarán y validarán las sesiones de usuario, y cómo se almacenarán de forma segura las contraseñas en la base de datos PostgreSQL. El sistema no requiere integración con proveedores externos (como Google u otros), por lo que se busca una solución de autenticación simple pero segura.

## Decisión tomada

Se utilizará **PyJWT** para la generación y validación de tokens de sesión bajo el estándar JWT (JSON Web Tokens), y **passlib** con el backend **bcrypt** para el hashing seguro de contraseñas al momento de registrarlas y verificarlas.

---

## Factores decisivos

1. El sistema solo requiere autenticación propia mediante usuario y contraseña con sesiones stateless basadas en tokens; no se necesita ninguna funcionalidad de OAuth, ni login con terceros, por lo que una solución liviana es la más adecuada.  
2. FastAPI provee un sistema de inyección de dependencias (`Depends`) que permite construir un verificador de tokens JWT reutilizable con muy poco código, haciendo que PyJWT sea suficiente para cubrir la necesidad sin frameworks más complejos.  
3. bcrypt es el algoritmo estándar de la industria para almacenar contraseñas hasheadas de forma segura; passlib ofrece una interfaz de alto nivel sobre bcrypt que simplifica su uso correcto.

## Alternativas

- **python-jose:** librería similar a PyJWT con soporte para un mayor número de algoritmos criptográficos (RSA, EC). Fue la opción recomendada en versiones anteriores de la documentación oficial de FastAPI, pero hoy en día PyJWT resulta más moderno y mejor mantenido. Además, dado que el sistema solo requiere el algoritmo HS256, el soporte adicional de algoritmos no representa un beneficio real.   
- **Authlib:** framework completo de OAuth 2.0 y OpenID Connect para Python. Sería la elección correcta si el sistema necesitara asociar el login con proveedores externos o exponer endpoints OAuth propios. Para un sistema de autenticación interna y básica como el de este proyecto, introduce una complejidad y una cantidad de configuración muy superiores a lo necesario.

## Pros y Contras de PyJWT y passlib con bcrypt

**Pros:**

- PyJWT es extremadamente liviano y hace exactamente una cosa: codificar y decodificar tokens JWT, sin efectos secundarios ni abstracciones innecesarias.  
- La combinación se integra de forma natural con el sistema `Depends()` de FastAPI.  
- passlib abstrae el uso correcto de bcrypt, reduciendo la probabilidad de errores de implementación.

**Contras:**

- Al no utilizar un framework de autenticación completo, funcionalidades como la revocación de tokens activos o el manejo de refresh tokens van a obviarse para este proyecto, y deberían implementarse manualmente si se quisieran en el futuro.

---

## Consecuencias

El componente será responsable de exponer al menos dos operaciones internas: generación de un token JWT al momento del login, y validación del token en cada request. Ambas APIs (REST y WebSockets) usarán esta lógica mediante el sistema de dependencias de FastAPI.   
Las contraseñas nunca se almacenarán en texto plano; únicamente el hash bcrypt será persistido en la base de datos.
