# ADR-008

---

## Título

Decisión de la base de datos persistente principal.

## Fecha

24/04/2026

## Estado

Tomada

---

## Contexto

El sistema debe persistir eventos de estacionamiento (cambios de estado), cuentas de usuario y datos históricos de ocupación para consultas a través de la API REST. Se debe elegir una tecnología de base de datos que sirva como almacenamiento persistente del sistema.

## Decisión

Usar **PostgreSQL** gestionado por AWS a través de RDS como base de datos relacional para todos los datos estructurados persistentes.

---

## Factores decisivos

1. Los datos del sistema son relacionales: los eventos referencian lugares, los lugares pertenecen a estacionamientos, los usuarios tienen roles, etc. Un modelo relacional es una opción natural.  
2. PostgreSQL es muy usado, está bien documentado y todo el equipo ya ha trabajado con él.  
3. Las consultas históricas complejas (por ejemplo, "ocupación promedio por lugar en los últimos 7 días" para calcular estadísticas) son sencillas a través de SQL sin necesidad de capas de consulta adicionales.  
4. RDS facilita el despliegue y la gestión de la base de datos significativamente.

## Alternativas

- **DynamoDB**: NoSQL gestionado de AWS con escalado serverless. Descartado porque el modelo de datos es relacional, y los patrones de acceso de DynamoDB son menos intuitivos para consultas históricas.  
- **MongoDB**: Almacén de documentos que ofrece flexibilidad de esquema. Descartado porque el esquema está bien definido y es relacional; la flexibilidad de un almacén de documentos no es necesaria para este tipo de datos y su falta de relaciones forzadas reduce las garantías de integridad de datos.  
- **SQLite**: Base de datos relacional embebida y liviana. Descartado porque no soporta escrituras concurrentes desde múltiples hilos/procesos de la aplicación en EC2.

## Pros y contras de PostgreSQL

**Pros:**

- Cumplimiento ACID completo asegura la integridad de los datos para los registros de eventos.  
- Capacidades de consulta SQL para análisis histórico.  
- Familiar para el equipo.

**Contras:**

- No escala horizontalmente sin configuración adicional significativa, aunque esto no es una preocupación a la escala del proyecto.
