# ADR-012

---

## Título

Despliegue del frontend.

## Fecha

16/03/2026

## Estado

Tomada

---

## Contexto

El sistema requiere un frontend web para que los usuarios visualicen la disponibilidad del estacionamiento en tiempo real y los datos históricos. El frontend se comunicará con el backend mediante REST (consultas históricas) y WebSocket (actualizaciones en vivo). Se debe seleccionar un framework para desarrollarlo y una plataforma para hostearlo.

## Decisión

Construir el frontend como una aplicación **Next.js** hosteada en los servidores de **Vercel**.

---

## Factores decisivos

1. Next.js provee una experiencia de desarrollo muy cómoda con enrutamiento basado en carpetas, y capacidades de server-side rendering.  
2. El modelo de despliegue de Vercel simplifica significativamente el pipeline de CI/CD del frontend.  
3. El nivel gratuito de Vercel es suficiente para las necesidades de tráfico del proyecto y elimina el costo de hosting para el frontend.  
4. Next.js es un framework ampliamente usado con documentación extensa, y parte del equipo ya es familiar con este, haciendo que la incorporación al proyecto sea directa.

## Alternativas

- **React simple (Vite/CRA) en AWS**: Mayor control pero requiere configuración adicional de AWS y complejiza ligeramente el desarrollo en comparación a next.js; todo sin agregar ventajas significativas sobre el hosteo en Vercel.  
- **Vue.js / Angular**: Alternativas válidas, pero el equipo tiene mayor familiaridad con React/Next.js.  
- **Frontend sin framework, puro html y js**: Se puede argumentar que otorga más flexibilidad, pero la complejidad extra al proceso de desarrollo no vale nada la pena. Descartado.

## Pros y contras de Next.js en Vercel

**Pros:**

- Despliegues sin configuración directamente desde Git.  
- Despliegues de "preview" a través de pull requests simplifican la revisión de código y el CI/CD.  
- Soporte integrado para variables de entorno, rutas de API y edge middleware.  
- Sin costo de hosteo del frontend.

**Contras:**

- Dependencia de Vercel para el frontend web.  
- Las conexiones WebSocket serán un poco más complejas de configurar.

---

## Consecuencias

Las conexiones WebSocket deben apuntar directamente al backend en EC2, ya que el modelo serverless de Vercel no soporta conexiones persistentes al servidor. Adicionalmente, será necesario crear un certificado TLS desde el lado del backend, ya que los navegadores no aceptan una conexión WebSocket sin certificado. Para esto seguramente se use Nginx con Let's Encrypt via Certbot.
